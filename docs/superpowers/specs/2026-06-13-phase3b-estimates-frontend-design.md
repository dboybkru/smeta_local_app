# Фаза 3b — Фронтенд редактора смет: дизайн

> Дизайн UI редактора смет поверх бэкенда фазы 3a. Опирается на общий дизайн-док `2026-06-11-smeta-app-design.md` (§6, §10) и спек `2026-06-12-phase3-estimates-design.md` (API/модель). Дизайн одобрен пользователем 2026-06-13 (в т.ч. через визуальные макеты).

## Рамки фазы 3b

Фронтенд для уже готового API смет (фаза 3a: клиенты, сметы, разделы, строки, итоги с роле-зависимой маржой). Две страницы: список смет и редактор сметы.

**Выбранные на брейншторме решения (с макетами):**
- **Макет редактора = вид A** — единая Excel-подобная таблица: разделы как группирующие заголовки, под ними строки позиций, итоги снизу. Ближе всего к текущим Excel-сметам пользователя; дизайн-док §10 требует «плотные таблицы в рабочих зонах»; проще на мобильном.
- **Добавление позиций = инлайн-поиск в таблице** — строка «+ добавить» внизу раздела: печать в ячейке наименования → выпадающий список из каталога → выбор добавляет строку (цены снапшотом с бэка) → курсор на «кол-во». Рядом кнопка «своя строка» для позиций не из каталога.
- **Архитектура «вид отделён от данных»** — стейт сметы (дерево + итоги + мутации) в хуке `useEstimate`, отдельно от компонента-представления (вид A). Это закладывает мультивид (B/C) на будущее без переделки логики.

**Отложено (фаза 3.5 / позже, заложено архитектурно):**
- Переключатель видов (B карточки / C двухпанельный) — выбор вида клиентом.
- Ветки-варианты (эконом/стандарт/премиум) — бэк уже создаёт одну «Базовую» ветку.
- Drag-drop сортировка строк/разделов (поле `sort_order` уже есть в API).
- Оптимистичные обновления UI.
- Экспорт Excel/PDF и публичные ссылки — фаза 4.

## Архитектура

React 19 + TypeScript + Vite + Tailwind v4 + react-router-dom v7, паттерны фазы 2b (api-модуль над `api()`, страницы, общий `AppHeader`, тесты vitest+RTL с мок-fetch).

**Ключевой принцип — представление отделено от данных:**
- `useEstimate(id)` грузит `GET /api/estimates/{id}` (полный `EstimateDetail`: ветки→разделы→строки + `totals`), хранит стейт, и предоставляет мутации. После каждой мутации — **reload сметы** (итоги пересчитывает бэк, всегда консистентны; оптимистичные апдейты отложены).
- Компонент представления (вид A — таблица) — отдельный слой, потребляющий `useEstimate`. Добавить вид B/C позже = новый компонент над тем же хуком.

**Решение: reload после мутаций (не оптимистичные апдейты).** Денежная логика (наценка→НДС→маржа) живёт только на бэке под TDD — один источник правды. Фронт НЕ дублирует расчёты; после правки перезапрашивает смету и показывает пересчитанные бэком итоги. Цена — лёгкая задержка/мигание; выигрыш — гарантированная корректность денег.

## Файлы

- `src/api/estimates.ts` — TS-типы (зеркало схем 3a: `Estimate`, `EstimateDetail`, `EstimateTotals`, `SectionDetail`, `LineDetail`, `Client`) + функции по эндпоинтам: `listEstimates`, `createEstimate`, `getEstimate`, `patchEstimate`, `deleteEstimate`, `addSection`, `patchSection`, `deleteSection`, `addLine`, `patchLine`, `deleteLine`, `listClients`, `createClient`.
- `src/hooks/useEstimate.ts` — `{ estimate, totals, loading, error, reload, canEdit, addSection, patchSection, deleteSection, addLine, patchLine, deleteLine, patchEstimate }`. `canEdit = user.role !== "viewer"` (через `useAuth`).
- `src/pages/EstimatesListPage.tsx` — список смет (`listEstimates`), форма создания (название, клиент, НДС вкл/ставка), переход в редактор, удаление.
- `src/pages/EstimateEditorPage.tsx` — редактор (вид A), оркестрирует хук + компоненты ниже.
- `src/components/estimate/EstimateHeader.tsx` — название (правка), клиент (выбор существующего / создать инлайн), НДС (вкл + ставка), статус (draft/sent/approved/archived).
- `src/components/estimate/SectionTable.tsx` + `LineRow.tsx` — таблица вида A: заголовок раздела (имя, наценка %, удалить), строки с инлайн-правкой `qty`/цен, удаление строки, строка-добавление.
- `src/components/estimate/CatalogSearchInput.tsx` — живой поиск по каталогу (debounced, переиспользует `listItems` из `src/api/catalog.ts` фазы 2b) с выпадающим списком; `onSelect(item)` → добавление строки.
- `src/components/estimate/EstimateTotalsBar.tsx` — плавающая панель итогов снизу: материалы / работы / без НДС / НДС / **Всего**; маржа — только если `totals.margin != null`.
- Модификации: `src/App.tsx` (роуты `/estimates`, `/estimates/:id` под `RequireAuth`), `src/components/AppHeader.tsx` (ссылка «Сметы»).

## Рабочий поток (вид A)

- **Список:** `/estimates` → таблица смет (название, клиент, статус, итог?). «Создать» → форма → `createEstimate` → переход в редактор. Удаление с подтверждением.
- **Редактор:** загрузка через `useEstimate`. Шапка правит смету (`patchEstimate`). «+ раздел» → `addSection`. В разделе: правка имени/наценки (`patchSection`), удаление (`deleteSection`).
- **Добавление позиции (инлайн-поиск):** в строке «+ добавить» печать → debounced `listItems(q)` → выпадающий список (наименование + цена) → выбор → `addLine({item_id, qty})` → reload. «Своя строка» → `addLine({name, unit, qty, work_price/material_price})`.
- **Правка строки:** инлайн-редактирование `qty` и цен (`patchLine` по blur/Enter) → reload. Удаление строки (`deleteLine`).
- **Итоги:** `EstimateTotalsBar` показывает `totals` из ответа; обновляется при каждом reload.

## Доступ

- `/estimates`, `/estimates/:id` под `RequireAuth`.
- **viewer = режим только-чтение** на фронте: `canEdit=false` → скрыты кнопки добавления/удаления, поля правки нередактируемы, маржа-колонка скрыта (бэк её и не отдаёт — `totals.margin == null`). Бэкенд остаётся барьером безопасности (403 на write), фронт — это UX поверх него.
- estimator видит/правит свои сметы; чужие в списке не появляются (бэк фильтрует), прямой переход на чужую → 404 → показать «не найдено».

## Тестирование

vitest + @testing-library/react, мок `fetch` (паттерн фазы 2b: `AuthProvider`-обёртка + `afterEach(cleanup)`):
- список смет рендерится; создание сметы → переход;
- редактор рендерит дерево (разделы/строки) + итоги из `totals`;
- инлайн-поиск каталога добавляет позицию (`addLine` вызван с `item_id`);
- правка `qty` шлёт `patchLine` и обновляет итоги (reload);
- маржа-колонка/итог скрыты когда `totals.margin == null`;
- viewer: режим только-чтение (нет кнопок правки).

## Решения (зафиксировано)

1. Вид A (Excel-подобная единая таблица) — единственный вид в MVP; мультивид заложен архитектурно, реализация отложена.
2. Добавление позиций — инлайн-поиск в таблице + «своя строка».
3. Стейт отделён от представления (`useEstimate` + компонент-вид).
4. Reload после каждой мутации (не оптимистичные апдейты) — один источник правды по деньгам на бэке.
5. viewer = режим только-чтение на фронте; маржа скрыта когда бэк вернул `null`.
6. Переиспользование `listItems`/каталожного API фазы 2b для поиска позиций.
