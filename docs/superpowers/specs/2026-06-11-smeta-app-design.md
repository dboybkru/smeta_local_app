# SmetaApp — дизайн системы

Дата: 2026-06-11
Статус: одобрен (брейншторм-сессия с владельцем)
Репозиторий: https://github.com/dboybkru/smeta_local_app.git
Продакшен-домен: **smetaapp.ru** (сервер 168.222.140.248, рядом с buhclaude, но изолированно)
Референс-проект: https://github.com/dboybkru/Smeta_app.git (старое приложение — источник проверенной логики, переписываем начисто)

## 1. Назначение

Веб-приложение для составления смет и коммерческих предложений на базе загружаемых прайсов поставщиков. Заменяет ручную работу в Excel. Многопользовательское, с прицелом на будущую монетизацию. Клиенты подрядчика получают КП по публичной ссылке без регистрации.

Эталон выходного документа — текущие Excel-сметы пользователя: шапка КП («Согласовано/Утверждаю»), разделы по объектам, таблица «№ / Наименование / Ед.изм. / Кол-во / Стоимость единицы (работы + материал) / Общая стоимость (всего, работы, материал)», итоги «Итого за оборудование / Итого работы / Всего по смете», блок подписей.

## 2. Стек и обоснование

| Слой | Выбор | Почему |
|---|---|---|
| Backend | FastAPI, Python 3.12 | Ядро продукта — обработка Excel (openpyxl/pandas), PDF (weasyprint), AI-пайплайны: сильнейшая экосистема в Python. Перенос проверенной логики из старого приложения |
| Frontend | React 19 + Vite + Tailwind + shadcn/ui | Лёгкий, быстрый, дизайн-система кастомизируется под фирменный стиль |
| Мобильность | PWA (манифест + service worker) | Установка на телефон и десктоп из браузера, офлайн-кэш просмотра. «Синхронизация с десктопом» = одно приложение везде, сервер — единый источник правды |
| БД | PostgreSQL 16 | Multi-tenant из коробки (tenant_id + RLS), JSONB, полнотекстовый поиск по каталогу, pgvector для AI-фич |
| HTTPS/прокси | Caddy | Автоматический Let's Encrypt для smetaapp.ru |
| Деплой | Docker Compose + GitHub Actions | Идентичность сред, тестовый гейт перед деплоем |

Отложено осознанно: local-first sync (PowerSync) — добавляется поверх Postgres позже, если реальная работа покажет потребность в полном офлайне; нативные мобильные приложения — PWA закрывает потребность.

│   │   ├── public/           # публичный доступ к КП по токену
│   │   └── ai/               # AI-ассистент (Claude API, tool use)
│   ├── tests/                # pytest; фикстуры — реальные сметы и прайсы
│   └── alembic/              # миграции
├── frontend/
│   └── src/
│       ├── pages/            # Сметы, Прайсы, Клиенты, Админ, Публичный просмотр
│       ├── components/
│       ├── api/              # типизированный клиент API
│       └── pwa/

Правила: модуль = одна зона ответственности; файлы не более ~300–400 строк; REST API с автодокументацией OpenAPI. Логика из старого `app.py` (222 КБ) переносится только через ревью и тесты, не копипастой. После каждой фазы обновляется graphify-граф проекта (`D:/git/graphify-out/`) для контроля архитектуры.

## 4. Модель данных

```
User            id, yandex_id?, email, password_hash?, name, role(admin|estimator|viewer),
                status(pending|active|blocked), created_at
PriceLevel      id, name, sort_order                  # «Закупка», «Опт», «Розница», «Партнёр»…
Supplier        id, name, column_mapping_template(JSONB)   # сохранённый маппинг колонок
PriceList       id, supplier_id, filename, imported_at, version
CatalogItem     id, supplier_id, name, article, unit, category, kind(material|work), search_tsv
ItemPrice       id, item_id, price_list_id, price_level_id, value   # N цен на позицию
Client          id, name, contacts, default_price_level_id
Estimate        id, client_id, owner_id, object_name, status, tax_mode, created_at
EstimateBranch  id, estimate_id, parent_branch_id?, name   # базовая + варианты
EstimateSection id, branch_id, name, sort_order, markup_percent
EstimateLine    id, section_id, item_id?, name, unit, qty,
                work_price, material_price, purchase_price_snapshot, sort_order
WorkRule        id, trigger(category|keyword), work_item_id, qty_formula
PublicLink      id, branch_id, token, expires_at?, watermark_enabled, watermark_text, revoked
AuditLog        id, user_id?, actor(user|ai), action, payload(JSONB), created_at
```

Ключевые решения:
- **Цены отдельно от позиций** (`ItemPrice`): любое число ценовых колонок, новый уровень — без миграции схемы.
- **Снапшот цен в строке сметы**: обновление прайса не меняет существующие сметы; разница с актуальным прайсом подсвечивается отдельно.
- **Маржа** = цена строки − `purchase_price_snapshot`; видна только авторизованным ролям, в экспорт и публичный просмотр не попадает.
- Полнотекстовый поиск по каталогу — `tsvector` + триграммы (поиск по части артикула).

## 5. Импорт прайсов

Поток: загрузка → распознавание → маппинг → предпросмотр → импорт.

1. Форматы v1: **xlsx, csv**. PDF и парсинг сайтов поставщиков — фаза 2 (перенос из старого приложения).
2. Распознавание: поиск строки заголовков, определение колонок; для многолистовых прайсов (Optimus, ~30 листов) — выбор листов, лист = категория.
3. Маппинг колонок файла на ценовые уровни системы; маппинг сохраняется в `Supplier.column_mapping_template` — повторный импорт того же поставщика в один клик.
4. Предпросмотр первых 20 строк с подсветкой проблем (пустые цены, дубли артикулов).
5. Версионность: импорт обновляет цены, прошлые версии хранятся; дельта цен видна («+12% с прошлого прайса»).

Прайс работ пользователя (`работы.xlsx`) — тот же механизм, позиции получают `kind=work`.

Эталонные входные файлы (тестовые фикстуры): прайсы Optimus, Bolid, «работы.xlsx»; сметы из `D:\Yandex.Disk\ИП\2026\…` (6 файлов).

## 6. Редактор смет

- Иерархия: смета → ветка → разделы (объекты) → строки. Соответствует структуре текущих Excel-смет.
- Добавление позиции: живой поиск по каталогу → цена по уровню клиента → количество. Произвольные строки не из каталога — можно.
- **Автоподбор работ**: при добавлении оборудования система предлагает работы по `WorkRule` (камера → «Монтаж камеры», то же количество); предложение подтверждается или отклоняется, правила настраиваются в админке.
- Итоги по разделу и смете (оборудование / работы / всего) — пересчёт на лету; корректировка процентом по разделу; режим налогообложения.
- **Ветки**: «Создать вариант» → копия, привязанная к базовой (эконом/стандарт/премиум); сравнение веток по итогам.
- Любая цена в строке переопределяется вручную; колонка «закупка/маржа» видна только владельцу сметы и админу.
- Мобильный режим (PWA): просмотр, правка количеств и статусов; полноценное редактирование заточено под десктоп.

## 7. Экспорт и публичные ссылки

- **Excel**: шаблон повторяет текущий формат КП пользователя (шапка, таблица с «в т.ч.», итоги, подписи, М.П.).
- **PDF**: тот же макет через weasyprint (HTML → PDF).
- **Публичная ссылка**: `https://smetaapp.ru/p/<токен>` — просмотр КП и скачивание PDF без регистрации. Настройки ссылки: срок жизни, отзыв, водяной знак (вкл/выкл, текст). Публичная версия не содержит закупочных цен и маржи.

## 8. Авторизация и роли

- Основной вход — **Яндекс ID (OAuth 2.0)**, redirect `https://smetaapp.ru/auth/yandex/callback`; для разработки — отдельное OAuth-приложение Яндекса с localhost-редиректом.
- Запасной вход email+пароль (и задел под монетизацию для не-яндекс-пользователей).
- Роли: `admin` (всё: пользователи, настройки AI, правила), `estimator` (свои сметы + каталог), `viewer` (просмотр).
- Новые регистрации → статус `pending` до одобрения админом.
- JWT (короткоживущий access + refresh). Все секреты — только в `.env`/переменных окружения. Захардкоженных паролей нет (исправление уязвимости старого репо, где дефолтный пароль админа лежал в коде).

## 9. AI-ассистент

- Чат в боковой панели редактора: «добавь 4 камеры 4Мп с монтажом и кабель 100 м» → ассистент через tool use ищет по каталогу, формирует черновик изменений, **пользователь подтверждает** — молчаливых правок нет.
- Claude API; инструменты: поиск по каталогу, добавление/изменение строк, расчёты итогов.
- Все действия AI пишутся в `AuditLog` с `actor=ai`.
- Фаза 2+: генерация сметы из описания объекта целиком, голосовой ввод.

## 10. Дизайн и стиль

- База — DESIGN_NOTES старого репо: спокойный редакторский стиль (референс nieuwbergen.com), светлая тема по умолчанию, тёплый нейтральный фон, тёмный текст, сдержанные зелёно-латунные акценты, serif-заголовки, тонкие границы, низкий визуальный шум. Тёмная тема — переключателем.
- shadcn/ui + Tailwind, токены палитры/типографики/отступов прорабатываются скилом `ui-ux-pro-max` на старте UI-фазы.
- Плотные таблицы в рабочих зонах (смета, прайс), воздух вокруг крупных областей. Операционные контролы не прячутся за декоративными макетами.
- Mobile-first; PWA: установка, иконка, сплеш, офлайн-кэш последних смет (просмотр).

## 11. Инфраструктура и деплой

- Docker Compose: `postgres`, `backend`, `frontend` (статика), `caddy` (авто-HTTPS).
- Сервер 168.222.140.248, каталог `/opt/smetaapp` — изолированно от buhclaude.
- CI/CD: GitHub Actions — пуш в `main` → тесты → сборка образов → деплой. Без прохождения тестов деплой не выполняется.
- Бэкапы: ежедневный `pg_dump`, хранение 30 дней.
- Локальная разработка: `docker compose -f docker-compose.dev.yml up` (Postgres + хот-релоад бэка и фронта).

## 12. Тестирование

- TDD для денежной логики: расчёты смет, импорт прайсов, ценовые уровни, маржа — тесты пишутся до кода.
- Фикстуры — реальные файлы пользователя (6 смет, 3 прайса): импорт каждого должен давать ожидаемые позиции и суммы; экспорт сметы должен совпадать с эталонной структурой.
- Перед мержем каждой фазы: `code-review` + `security-review`; ручная проверка UI через `verify`.

## 13. Фазы реализации

1. **Каркас**: репозиторий, Docker, Postgres, миграции, скелеты модулей, CI; авторизация (Яндекс + email, роли, pending-аппрув).
2. **Каталог**: модели цен/уровней/поставщиков, импорт xlsx/csv с маппингом и версиями, поиск.
3. **Сметы**: редактор, разделы/строки/итоги, ветки, автоподбор работ, клиенты и уровни цен.
4. **Экспорт**: Excel/PDF по эталонному шаблону, публичные ссылки с водяными знаками.
5. **AI-ассистент**: чат, tool use, аудит.
6. **PWA и полировка**: манифест, service worker, мобильные сценарии, деплой на smetaapp.ru.

Каждая фаза завершается: рабочим продуктом, зелёными тестами, ревью, обновлением graphify-графа.
