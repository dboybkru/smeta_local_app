# Инвайт по email-ссылке (invite-only онбординг) — дизайн

**Дата:** 2026-06-16
**Статус:** на ревью пользователя (брейншторм)
**Очередь:** A (jobs) → B (httpOnly-cookie+CSRF) → C (этот спек). Зависит от B (страница приёма ставит cookie-сессию).

## Проблема

Онбординг Этапа A: org_admin/superuser приглашает по email → создаётся `User(status="invited")`, человек «забирает» строку при регистрации/Яндекс-входе. Но: (1) письмо НЕ шлётся (отложено), (2) **формы регистрации в UI нет**, (3) Яндекс не подходит для не-Яндекс-почты (напр. `@bk.ru`). Итог: приглашённый **физически не может войти** — инвайт это тупик. Подтверждено на проде: `dboy@bk.ru` висит `invited` без возможности claim.

## Решение (принято на брейншторме)

**Invite-only онбординг по токен-ссылке из письма.** При инвайте генерится одноразовый токен со сроком, шлётся письмо со ссылкой; приглашённый открывает страницу приёма, ставит имя+пароль → активен в назначенной орг/роли → залогинен (cookie из B). Открытой саморегистрации нет (только бутстрап первого юзера). Транспорт письма — **SMTP-релей** (есть ящик), креды шифрованно в `app_settings` (как DaData/Яндекс).

## Модель / миграция
На `User` (invited-строка) добавить:
- `invite_token: str | None` (индекс), `invite_expires_at: datetime | None`.
Миграция `down_revision =` миграция из A (цепочка A→C; B без миграции). Колонки nullable (нужны только для invited).

## Backend

### Отправка почты — `app/email/`
- `EmailSender.send(to, subject, html, text)` — интерфейс.
- SMTP-реализация (`smtplib`/`aiosmtplib`), конфиг из `app_settings`: `smtp_host`, `smtp_port`, `smtp_user`, `smtp_password` (Fernet), `smtp_from`, `smtp_tls`. Нет конфига → `EmailNotConfigured`.
- `app/settings`: `GET/PUT /api/settings/smtp` (superuser), пароль write-only; UI в `/admin/ai` рядом с DaData/Яндекс.

### Инвайт (расширение существующего)
`POST /api/orgs/{org_id}/users` — после создания invited-строки: генерит `invite_token=token_urlsafe(32)`, `invite_expires_at=now+7д`, шлёт письмо со ссылкой `{frontend_url}/invite/{token}`. **Graceful:** ошибка/отсутствие SMTP не валит инвайт — строка создаётся, ответ `{...user, email_sent: bool, invite_link: str}`. Админ видит ссылку (передать вручную, если письмо не ушло).
`POST /api/orgs/{org_id}/users/{uid}/resend-invite` (новый) — перевыпуск токена+срока, переотправка письма; 404 если юзер не invited/не в орг.

### Приём (claim по токену)
- `GET /api/auth/invite/{token}` → `{email, org_name, role}`; 404 если токена нет/юзер не invited, 410 если просрочен.
- `POST /api/auth/invite/{token}/accept` `{name, password}` → ставит `password_hash`, `name`, `status=active`, гасит токен/срок → сессия (cookie из B) → `UserOut`. Повторный токен/просрочка → 404/410.

### Регистрация — invite-only
`POST /api/auth/register`: если в системе НЕТ юзеров → создаёт первого (superuser+active, бутстрап) и логинит. Иначе → **403** «Регистрация только по приглашению». Claim приглашённых идёт через токен-эндпоинт, НЕ через open register. (Существующие онбординг-тесты обновить: «саморегистрация без инвайта» теперь 403; «claim invited» — через токен-флоу.)

## Frontend
- Роут `/invite/:token` + `InvitePage`: на mount `GET invite/{token}` → показывает «Вас пригласили в {org} как {role}», форма имя+пароль → `accept` → внутрь (редирект `/`). Просрочка/неверный токен → понятная плашка.
- `/login`: без ссылки «регистрация» (invite-only); подсказка «нет аккаунта — попросите приглашение у администратора».
- Инвайт-форма (OrgsPage / AdminUsersPage): показывает `invite_link` если `email_sent=false` (копировать) + кнопка «Переотправить приглашение» для invited-юзеров.
- SMTP-секция в `/admin/ai` (host/port/user/password/from/tls), superuser-only.

## Тестирование
- Backend: инвайт генерит токен+срок и зовёт EmailSender (замокан); `email_sent=false` + `invite_link` при незаданном SMTP (инвайт не падает); `GET invite` (ок/404/410); `accept` активирует и гасит токен, повтор → 404; open register 403 когда юзеры есть, ок для первого; resend перевыпускает токен. EmailSender замокан (без сети). SMTP-настройки CRUD (superuser).
- Frontend: InvitePage (загрузка инфо, accept, просрочка); login без «регистрации»; инвайт-форма показывает ссылку/переотправку.

## Пред-условие (ops, на пользователе)
Доставляемость: SPF/DKIM/DMARC на отправляющем домене `smetaapp.ru` (иначе письма в спам) + SMTP-креды ящика. Без настроенного SMTP фича деградирует к показу `invite_link` админу (не ломается).

## Вне рамок
Транзакционный email-API (только SMTP-релей сейчас); шаблонизатор писем (одно простое HTML+text-письмо); множественные активные инвайты на email (одна invited-строка = один токен); самостоятельная смена email/пароля пользователем.
