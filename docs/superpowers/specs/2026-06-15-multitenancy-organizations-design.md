# Мультиарендность SmetaApp — Этап A: Организации (изоляция данных) — дизайн

**Дата:** 2026-06-15
**Статус:** на ревью пользователя (брейншторм)
**Очередность:** A (организации, этот спек) → B (направления/команды внутри орг) → C (публикация прайсов между орг)

## Цель

Превратить приложение из одно-арендного в мульти-арендное: данные каждой **организации**
(фирмы/направления) полностью изолированы — сметы, каталог, прайсы, поставщики, уровни цен,
клиенты, реквизиты не видны другим организациям. Иерархия ролей: сквозной **суперюзер** →
**админ организации** (раздаёт доступы внутри своей орг) → **сметчик** → **viewer**.

Покрывает сценарии: (2) несколько фирм в одном инстансе, не видят данные друг друга; и
(1-вариант) «направление = отдельная организация со своим админом». Направления, делящие общий
каталог фирмы (под-команды), — Этап B. Публикация прайсов между орг — Этап C.

## Контекст (текущее состояние)

Одно-арендная модель: `User`(role admin/estimator/viewer, status), `Estimate.owner_id`
(сметчик видит свои, admin — все). **Глобальны** (общие для всех): `CatalogItem`, `Supplier`,
`PriceList`, `PriceLevel`, `ItemPrice`, `Client`, `CompanyProfile`(по пользователю), AI-слой,
`AppSetting`(DaData/Яндекс). Прод: одна реальная фирма с данными + админ `daniil.gurov`.

## Принятые решения (брейншторм)

1. **Организация** — граница изоляции (своё всё). Гибкость «общий прайс vs свой» достигается так:
   общий прайс → под-команды в одной орг (Этап B); свой прайс → отдельные орг (Этап A).
2. **Онбординг** — инвайт по email через предсоздание строк + claim при входе (без рассылки писем).
3. **Видимость смет** (до Этапа B): сметчик — свои; org_admin — все сметы своей орг.
4. **AI/DaData/Яндекс** — глобально (настраивает суперюзер). Без `org_id`.
5. **CompanyProfile** (реквизиты для КП) — становится **на организацию** (одна на орг).
6. **Роли:** `is_superuser`(флаг, сквозной) + `role` в пределах орг: `org_admin / estimator / viewer`.
7. **Миграция:** дефолтная орг + backfill `org_id` на все строки; существующие `role="admin"` →
   `is_superuser=true` + `role="org_admin"`.

# Этап A — детальный дизайн

## A1. Модель данных

**Новая таблица `organizations`** (`app/orgs/models.py`):
- `id` PK, `name: str(255)` (unique), `created_at: datetime(tz)`.

**`User`** (`app/auth/models.py`) добавляются:
- `org_id: int | None` FK→organizations.id, index (nullable: org-less самрегистрации и до привязки).
- `is_superuser: bool` (server_default false).
- `role` значения теперь: `org_admin / estimator / viewer` (старое `admin` мигрируется в `org_admin`).
- `status` добавляет значение `invited` (предсозданный, ещё не вошёл) к `pending/active/blocked`.

**`org_id: int` (NOT NULL) добавляется на:** `Estimate`, `Client`, `CatalogItem`, `Supplier`,
`PriceList`, `PriceLevel`, `CompanyProfile`. Каждый — FK→organizations.id, index.
Дочерние наследуют орг через родителя, свой `org_id` им НЕ добавляем: `ItemPrice`(через item/list),
`EstimateBranch/Section/Line`(через estimate), `AssistantMessage`/`PublicLink`(через estimate).

**Изменения уникальностей (в миграции):**
- `Supplier.name`: глобальный unique → unique `(org_id, name)`.
- `PriceLevel.name`: глобальный unique → unique `(org_id, name)`.
- `CatalogItem` UC `(supplier_id, article, name)` — остаётся (supplier уже орг-скоупит).
- `CompanyProfile`: было «одна на пользователя» → станет «одна на орг», unique `(org_id)`.
- `User.email`, `User.yandex_id` — остаются глобально уникальными (один человек = один аккаунт).

**Глобальны (без `org_id`):** `AIProvider/AIModel/AIPurpose/AIUsage`, `AppSetting`, `Job`
(но extract-job скоупится параметром org_id — см. A6).

## A2. Роли и права

| Действие | superuser | org_admin | estimator | viewer |
|---|---|---|---|---|
| Создавать/переименовывать организации | ✅ | — | — | — |
| Управлять пользователями орг (инвайт, роль, блок) | ✅ любой орг | ✅ своей орг | — | — |
| Каталог/поставщики/прайсы/уровни своей орг — запись | ✅ | ✅ | — | — |
| Каталог/клиенты/уровни своей орг — чтение | ✅ | ✅ | ✅ | ✅ |
| Реквизиты орг (CompanyProfile) | ✅ | ✅ | — | — |
| Клиенты — создание/правка | ✅ | ✅ | ✅ | — |
| Сметы — свои (создание/правка) | ✅ | ✅ | ✅ | — |
| Сметы — все в орг (просмотр/правка) | ✅ | ✅ | — | — |
| AI/DaData/Яндекс (глобально) | ✅ | — | — | — |

`viewer` — чтение; в Этапе A видит сметы по правилу видимости (свои/выданные), полную ценность
получает в Этапе B (гранулярная выдача). Суперюзер по данным работает в рамках СВОЕЙ орг
(дефолтной); чужие орг администрирует через `/api/orgs/...` (управление, не просмотр данных).

## A3. Изоляция доступа (механизм)

- Зависимость `current_org(user) -> int`: возвращает `user.org_id`; если `None` → 403
  «Пользователь не привязан к организации».
- **Все** списки/гет/мутации скоупятся: `WHERE org_id == current_org`. Доступ к строке чужой орг →
  **404** (как сейчас с чужими сметами — не раскрываем существование).
- Реализация: в каждом `service.search_*/get_*/list_*` добавляется фильтр по `org_id`; при создании
  строки проставляется `org_id = current_org`. Существующая `get_owned_estimate` сначала проверяет
  `est.org_id == user.org_id` (иначе 404), затем правило «estimator — только свои».
- `require_admin` (глобальный) → переименовать смысл: для орг-операций ввести `require_org_admin`
  (роль org_admin ИЛИ is_superuser); для глобальных (AI/настройки) — `require_superuser`.

## A4. Онбординг (инвайт по email + claim)

- **Инвайт:** org_admin/суперюзер «Добавить пользователя» → email + роль → создаётся
  `User(email, org_id, role, status="invited")` без `password_hash`/`yandex_id`.
- **Claim при входе:**
  - **Email-регистрация** с этим email: если есть строка `status="invited"` без пароля →
    проставить пароль+имя, `status="active"`. Если строка уже `active` → 409.
  - **Яндекс-вход** с этим email: если есть `invited`-строка → привязать `yandex_id`,
    `status="active"`. Иначе обычная логика.
- **Самрегистрация без инвайта** (email/Яндекс, email никем не приглашён): создаётся
  `User(status="pending", org_id=null)` — доступа к данным нет (403 в `current_org`), пока
  суперюзер/админ не привяжет к орг и не активирует. Открытая регистрация остаётся, но бесполезна
  без привязки — защита от случайного доступа.
- **Первый пользователь системы** (как сейчас «первый = активный админ») — на проде уже есть; для
  чистой установки: первый зарегистрированный → `is_superuser=true`, создаёт первую орг.

## A5. Управление организациями (суперюзер)

- `GET /api/orgs` — список организаций (id, name, кол-во пользователей).
- `POST /api/orgs` `{name}` — создать; 409 при дубле имени.
- `PATCH /api/orgs/{id}` `{name}` — переименовать.
- `GET /api/orgs/{id}/users`, `POST /api/orgs/{id}/users {email, role}` (инвайт),
  `PATCH /api/orgs/{id}/users/{uid} {role?, status?}` (роль/блок), — суперюзер для любой орг.
- org_admin управляет пользователями СВОЕЙ орг через те же ручки, но `{id}` обязан совпадать с его
  орг (иначе 403). Удаление пользователей не делаем (блокировка `status="blocked"`).

## A6. Прочие скоуп-правки

- **Каталог-импорт/фасеты/поиск** (`catalog/service`, `router`): фильтр по `org_id`; импорт ставит
  `org_id` на новые `CatalogItem/Supplier/PriceList/PriceLevel`.
- **Extract-job:** `start_extract_characteristics` пишет `org_id` в `Job.params`; воркер
  `extract_batch(..., org_id=)` фильтрует `CatalogItem.org_id == org_id` — извлечение не трогает
  чужие орг.
- **CompanyProfile API:** `GET/PUT /api/profile` → реквизиты ОРГ (одна на орг), правит org_admin;
  estimator читает (для генерации КП). Экспорт/КП берут профиль орг текущей сметы.
- **Клиенты:** `org_id` на Client; suggest (DaData) — глобальная интеграция, но сохранённые клиенты
  орг-скоупятся.

## A7. Миграция (Alembic, проверить на Postgres)

Одна миграция от текущего head `e2f3a4b5c6d7`:
1. `create_table organizations`.
2. Вставить дефолтную орг (id фиксирован, имя из env `DEFAULT_ORG_NAME` или «Организация»).
3. `add_column org_id` (nullable) на все целевые таблицы; `UPDATE ... SET org_id = <default>`;
   затем `alter_column org_id NOT NULL` (на User — оставить nullable).
4. `User`: add `is_superuser`(false), backfill `is_superuser=true, role='org_admin'` где `role='admin'`.
5. Перестроить уникальности: drop глобальных unique на `suppliers.name`, `price_levels.name`,
   `company_profiles` (per-user); создать unique `(org_id,name)` / `(org_id)`.
6. `CompanyProfile`: backfill `org_id`; если несколько профилей — оставить «первый» на орг
   (остальные удалить или перенести владельца — для прода профиль один).
Boolean `server_default=sa.text('false')`. Все целевые модели регистрируются в conftest.

## A8. Фронтенд

- **Суперюзер:** страница `/admin/orgs` — список/создание организаций, у каждой — управление
  пользователями (инвайт по email + роль, смена роли, блок).
- **org_admin:** существующая «Пользователи» — в рамках своей орг + «Пригласить по email».
- В шапке (`AppHeader`) — название текущей организации; пункт «Организации» виден только суперюзеру.
- Остальной UI без изменений — изоляция обеспечивается бэкендом.

## A9. Обработка ошибок / краевые случаи

- org-less пользователь (pending без орг) → 403 на всех данных, понятная плашка на фронте
  «Аккаунт не привязан к организации — обратитесь к администратору».
- Попытка доступа к чужой орг → 404.
- Инвайт на уже существующий active-email → 409.
- Суперюзер без `org_id` (теоретически) → данные-эндпоинты 403; орг-управление работает.

## A10. Тестирование (фокус — изоляция)

- **Изоляция:** юзер орг A не видит и не правит сметы/каталог/прайсы/уровни/клиентов/профиль орг B
  (404); поиск/списки возвращают только свою орг; импорт пишет в свою орг.
- **Онбординг:** claim invited-строки email-регистрацией и через Яндекс; самрегистрация без инвайта →
  pending/org-less → 403; дубль-инвайт → 409.
- **Роли:** estimator не управляет пользователями (403); org_admin не лезет в чужую орг (403);
  суперюзер создаёт орг и инвайтит в любую.
- **Миграция:** на Postgres — backfill всех `org_id`, NOT NULL, per-org уникальности, admin→superuser.

# Дорожная карта (зафиксировано, не в этом спеке)

## Этап B — Направления/Команды внутри организации
Опциональный под-уровень: `Team`(org_id, name, lead_user_id); членство пользователей в командах;
сметы могут скоупиться к команде; лид раздаёт доступ к сметам внутри команды (гранулярная выдача —
`EstimateAccess(estimate_id, user_id, level)`). `viewer` обретает полную ценность. Решает «направления
делят общий каталог фирмы, но сметы разграничены, у направления свой главный».

## Этап C — Публикация прайсов между организациями + аудитория-цены
- `CatalogPublication`(source_org_id, scope=каталог/прайс-лист, exposed_price_level_id, audience).
  Audience = список целевых орг ИЛИ код-приглашение, который целевая орг активирует.
  Пример Optimus: две публикации своего каталога — на уровне «Партнёр» для партнёрских орг и
  «Монтаж» для монтажных орг.
- Целевая орг видит опубликованные внешние каталоги **только на чтение по выданному уровню**;
  добавляет позиции в свои сметы со снапшотом этой цены; закупку/другие уровни источника не видит.
- **Монтажник со своими ценами** (внутри орг): `User.default_price_level_id` +
  `User.default_markup_percent`; смета по умолчанию берёт настройки автора; клиенту уходит цена
  монтажника, закупка/маржа скрыты (механизм наценки/скрытия уже есть).
- **Forward-compat:** Этап A это не блокирует — каталог и уровни цен уже на организацию; публикация
  и аудитория-цены — слой выдачи поверх, без переделки A.

## Вне рамок (YAGNI для A)
Биллинг/лимиты по орг; переключатель «текущая орг» для сквозного просмотра данных суперюзером;
токен-инвайты по email-ссылке (нужна рассылка); per-org AI/DaData-ключи; удаление пользователей/орг.
