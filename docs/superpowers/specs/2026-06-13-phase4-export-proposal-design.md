# Фаза 4 — Экспорт, КП и публичные ссылки: дизайн

> Уточняющий спек к общему дизайн-доку `2026-06-11-smeta-app-design.md` (§7). Дизайн одобрен пользователем 2026-06-13.

## Рамки и разбивка

**Разбивка:** 4a — бэкенд (модели, AI-генерация КП, рендеринг Excel/PDF/HTML, публичные ссылки); 4b — фронтенд (профиль, редактор блоков КП, кнопки экспорта, управление публичными ссылками, публичная страница). Как фазы 2/3.

**Цель:** каждая смета превращается в **коммерческое предложение (КП)** с тремя уровнями вывода, которое можно скачать (Excel/PDF) или отдать клиенту публичной ссылкой. Маркетинговые тексты КП по умолчанию **генерирует AI** из сметы и профиля исполнителя; пользователь их редактирует или удаляет.

**Два разных слоя документа:**
- **Смета/спецификация** — таблица разделов→позиций с итогами (готова из фазы 3).
- **Маркетинговые блоки КП** — заголовок, боль, решение, УТП, преимущества, условия, контакты, CTA (тексты; AI-генерация + ручная правка).

**Три уровня вывода** (`level`): `full` (полное КП: маркетинг + смета), `cover` (титул + смета), `estimate` (только смета-таблица). Выбираются при экспорте/публикации.

**Отложено (фаза 5 / позже):** интерактивный AI-ассистент редактора (чат, tool use, добавление позиций) — здесь только узкий генератор текстов КП; ветки-варианты в публичных ссылках (привязка к estimate, не branch).

## Модель данных (4a)

```
CompanyProfile  id, user_id (FK→users, unique), org_name, inn, contacts (JSONB:
                {phone,email,address,site}), bank_requisites (text),
                utp (JSONB: list[str]), cases (JSONB: list[str]), guarantee (text),
                logo_url? , updated_at
                # реквизиты исполнителя + переиспользуемые блоки; один на пользователя
```

`Estimate` расширяется одним полем:
```
Estimate.proposal  JSONB nullable  # блоки КП:
  { "title": str, "subtitle": str, "pain": str, "solution": str,
    "advantages": [str], "terms": str, "cta": str }
  # все блоки опциональны; null = КП ещё не сформировано
```

```
PublicLink      id, estimate_id (FK→estimates), token (unique, urlsafe),
                level (full|cover|estimate, default full),
                expires_at? (datetime), watermark_enabled (bool, default false),
                watermark_text (str, default ""), revoked (bool, default false),
                created_at
```

Деньги — по-прежнему Decimal/строки. Миграция Alembic — **вручную, проверять на Postgres** (boolean `server_default` = `false`, не `0` — урок фазы 3a).

## AI-генерация КП (Claude API)

- Сервис `generate_proposal(db, estimate, profile) -> dict` — формирует промпт из позиций сметы (наименования, категории, суммы), объекта, профиля (УТП/кейсы/гарантия) и просит Claude вернуть JSON блоков (title/subtitle/pain/solution/advantages/terms/cta) на русском, в тоне продающего КП (структура: заголовок-выгода, боль клиента, решение-результат, УТП, преимущества, условия, CTA).
- Эндпоинт `POST /api/estimates/{id}/proposal/generate` (auth, владелец) → вызывает сервис, пишет результат в `Estimate.proposal`, возвращает его. Идемпотентно перезаписывает (с подтверждением на фронте).
- `PATCH /api/estimates/{id}/proposal` — ручная правка блоков (partial); передача `null`/`{}` очищает.
- Claude API: ключ `ANTHROPIC_API_KEY` в `.env` (config настройка, опциональна — без ключа эндпоинт generate возвращает 503 «AI не настроен», ручное заполнение работает). Модель и параметры — **сверить по `claude-api` skill при реализации** (актуальные id моделей; вероятно `claude-haiku-*` для дешёвой генерации текста). Вызов с таймаутом; ошибки Claude → 502/503, не 500.
- Тесты: Claude API **замокан** (не дёргать реальный API в тестах) — проверяется построение промпта и запись результата в proposal.

## Рендеринг — один HTML-шаблон, три формата

Один Jinja2-шаблон `proposal.html` рендерит КП по `level`:
- `full` — маркетинговые блоки (из `proposal`) + шапка профиля + таблица сметы + итоги + условия + подпись/М.П.
- `cover` — титул (заголовок/подзаголовок/контакты) + таблица сметы + итоги.
- `estimate` — только шапка-реквизиты + таблица сметы + итоги + подпись/М.П.

Из шаблона:
- **Публичная HTML-страница** — рендер шаблона как есть (адаптив, мобильно).
- **PDF** — `weasyprint` рендерит тот же HTML в PDF.
- **Excel** — отдельный генератор `openpyxl`: шапка реквизитов + таблица разделов/позиций с «в т.ч.» (материалы/работы) + итоги с НДС + место под подпись/М.П. По уровню `full`/`cover` добавляются текстовые блоки на отдельный лист/сверху; `estimate` — только таблица. (openpyxl уже в проекте — импорт каталога.)

**Стиль КП:** спокойный редакторский (дизайн-док §10) — serif-заголовки, светлый фон, плотная таблица сметы, сдержанные акценты.

## API (4a)

Под `require_active`, владелец сметы (estimator свои / admin все):
| Метод | Путь | Назначение |
|---|---|---|
| GET / PUT | `/api/profile` | профиль исполнителя текущего пользователя |
| POST | `/api/estimates/{id}/proposal/generate` | AI-генерация блоков КП |
| PATCH | `/api/estimates/{id}/proposal` | ручная правка/очистка блоков |
| GET | `/api/estimates/{id}/export.xlsx?level=` | скачать Excel |
| GET | `/api/estimates/{id}/export.pdf?level=` | скачать PDF |
| POST | `/api/estimates/{id}/public-links` | создать ссылку `{level, expires_at?, watermark_enabled?, watermark_text?}` |
| GET | `/api/estimates/{id}/public-links` | список ссылок сметы |
| DELETE | `/api/public-links/{id}` | отозвать (revoked=true) |

**Публичные (БЕЗ auth)** — отдельный роутер:
| GET | `/p/{token}` | HTML-страница КП по уровню ссылки |
| GET | `/p/{token}/pdf` | PDF КП |

Публичный доступ: токен не найден/отозван → 404; просрочен (`expires_at` в прошлом) → 410. **Публичный вывод НИКОГДА не содержит закупочных цен и маржи** (рендер использует только продажные суммы/итоги; маржа/закупка вырезаны на уровне формирования контекста шаблона). Водяной знак (если `watermark_enabled`) накладывается на HTML/PDF.

## Тестирование (TDD/денежная логика уже покрыта фазой 3a)

- Профиль: GET/PUT, один на пользователя.
- proposal: generate (Claude замокан) пишет блоки; PATCH правит/очищает; доступ только владелец.
- Excel-генерация: возвращает валидный xlsx (openpyxl читает обратно), есть шапка/позиции/итоги; уровни.
- PDF-генерация: возвращает непустой PDF (`%PDF` сигнатура), 200, корректный content-type.
- Публичные ссылки: создание/список/отзыв; публичный GET валидного токена → 200; отозванного → 404; просроченного → 410; **в публичном HTML/контексте нет полей маржи/закупки** (проверка по содержимому); водяной знак присутствует когда включён.
- Экспорт/публичный доступ к чужой смете: estimator не может экспортировать чужую (404).

## Решения (зафиксировано)

1. Разбивка 4a (бэкенд) / 4b (фронтенд); 4a крупная — дробится на задачи, при необходимости делится надвое.
2. `proposal` — JSONB-поле на `Estimate` (гибкий состав блоков, без миграций при изменении набора).
3. `CompanyProfile` — один на пользователя.
4. `PublicLink` привязан к `estimate_id` (ветки отложены).
5. AI-генерация текстов КП в фазе 4 (Claude API, замокан в тестах, ключ опционален); интерактивный ассистент — фаза 5.
6. Один HTML-шаблон → публичная страница + PDF (weasyprint); Excel — отдельный openpyxl-генератор.
7. Три уровня вывода `full`/`cover`/`estimate`.
8. Публичный вывод без закупки/маржи; токен с проверкой срока/отзыва; водяной знак.
9. Новые зависимости: `weasyprint`, `anthropic` (или httpx к Claude API). Миграции — вручную, проверка на Postgres.
