# Надёжный парсер прайс-листов (авто-определение структуры) — дизайн

**Дата:** 2026-06-14
**Статус:** одобрен пользователем (брейншторм)

## Цель

Импортировать прайс-листы разных поставщиков с разной структурой одним предсказуемым
механизмом: парсер сам определяет строку заголовка и раскладку колонок по их подписям,
понимает двухстрочные шапки, пустые колонки-спейсеры, строки-категории и «цену по запросу».
«Выбрать все листы» у многолистовых прайсов (Optimus, pricetin) должно работать без порчи данных.

## Контекст и проблема

Текущий импортёр применяет **одну карту колонок по номерам** ко всем выбранным листам и берёт
**одну строку заголовка**. Это ломается на реальных прайсах: листы с разной раскладкой получают
сдвиг колонок → цена попадает в характеристики, комментарий — в цену (тихая порча данных).

Проанализированы 6 реальных файлов (`D:\git\прайсы\`):

| Файл | Листов | Шапка | Особенности |
|---|---|---|---|
| bolid_price.xlsx | 1 | строка 1 | плоский; есть Код и Артикул; 2 цены (розн/опт) |
| работы.xlsx | 1 | строка 1 | работы (kind=work); Наименование+Цена+Ед.изм; 1 цена |
| Прайс-лист Контроль доступа.xlsx | 1 | строки 2+3 | колонка «Производитель»; артикула нет; цены «звоните»; строки-категории (№ 1.1.1) |
| Прайс-лист Охранно-пожарная сигнализация.xlsx | 1 | строки 2+3 | та же схема |
| 02.06.26_Price_Optimus….xlsx | 24 | строки 4/6/7 | 3 разные раскладки; на части листов нет пустой колонки C → сдвиг |
| pricetin.xlsx | 21 | строки 5+6 | пустая колонка A (спейсер); ед.изм в «руб./шт»; строки-категории; единая раскладка на всех листах |

Сквозные сложности: плавающая строка заголовка; пустые колонки-спейсеры; двухстрочные шапки
(уровни цен подписаны «1/2/3» под объединённой «Цены»); строки-категории вперемешку с товарами;
нечисловые цены («звоните»); ед.изм внутри текста («руб./шт»); разные колонки артикула (Код/Артикул/нет);
колонка «Производитель»; цены в у.е.

## Принятые решения (брейншторм)

1. Подход — **авто-определение по названиям колонок** + подтверждение в мастере (не ручная карта,
   не захардкоженные профили поставщиков).
2. **Производитель** → отдельное поле `CatalogItem.manufacturer`.
3. **Категория** → из ближайшей строки-категории сверху (захват `current_category`).
4. **Ключ товара (upsert)** = Артикул производителя, при отсутствии — Код поставщика, иначе имя.
5. **Цена в у.е. / «звоните» / «по запросу» / «уточняйте»** → цена 0 + отметка
   **«уточнить стоимость»** на уровне **всей позиции** (`CatalogItem.price_on_request`).

## Архитектура и компоненты

### 1. Движок определения — `backend/app/catalog/detect.py` (новый)

```python
@dataclass
class PriceColumn:
    index: int        # абсолютный индекс колонки
    label: str        # подпись ("РОЗН.", "Розничная_цена", "Цена 1"…)
    sample: str       # пример значения из первой строки данных
    on_request: bool  # колонка в у.е. (заголовок содержит "у.е."/"усл.")

@dataclass
class DetectedLayout:
    header_row: int            # 0-индекс строки заголовка
    data_start_row: int        # 0-индекс первой строки данных
    name_col: int | None
    article_col: int | None    # Артикул, иначе Код
    chars_col: int | None
    unit_col: int | None
    manufacturer_col: int | None
    price_columns: list[PriceColumn]
    confidence: float          # доля уверенности (для подсветки в мастере)

def detect_layout(rows: list[list]) -> DetectedLayout | None: ...
```

**Поиск строки заголовка:** сканируем первые 20 строк; для каждой считаем число ячеек,
нормализованный текст которых совпал со словарём ролей. Строка-заголовок = максимум совпадений,
обязательно содержит роль «Наименование» и хотя бы одну ценовую. При равенстве — самая ранняя.
Если такой строки нет → `None` (мастер попросит указать вручную).

**Нормализация ячейки:** `str(c).strip().lower()`, схлопывание пробелов, замена `ё→е`.

**Словарь синонимов (по приоритету внутри роли):**
- name: `наименование работы`, `наименование`, `название`, `товар`
- article: `артикул` → `код товара` → `код`  (Артикул важнее Кода)
- chars: `краткая характеристика`, `краткие характеристики`, `характеристики`, `характеристика`, `описание`
- unit: `ед. изм`, `ед.изм`, `единица`, `вал./ед. изм`, `ед`
- manufacturer: `производитель`, `бренд`, `вендор`
- price (одиночные): `розничная_цена`/`розн`, `оптовая_цена`/`опт`, `кр.опт`, `парт`, `инст`, `цена`, `стоимость`

**Назначение ролей:** по каждой колонке строки заголовка матчим текст против словарей; первая
непустая роль закрепляется за колонкой. Роль закрепляется один раз (первое совпадение слева).

**Двухстрочные шапки (Цены 1/2/3):** если ячейка заголовка нормализуется к `цены`/`цены руб.`/`цены, руб.`
(общее множественное, без конкретного уровня), то реальные ценовые колонки берём из строки
`header_row+1`: непустые ячейки от индекса «Цены» вправо → `PriceColumn` с `label` из под-значения
(«1»/«2»/«3» → «Цена 1/2/3»). В этом случае `data_start_row = header_row + 2`. Иначе ценовые
колонки — одиночные совпадения по ценовым словам, `data_start_row = header_row + 1`.

**Пустые колонки-спейсеры** (pricetin A, Optimus C) обрабатываются автоматически: раскладка по
абсолютному индексу совпавшего заголовка, а не по порядку следования.

**у.е.-колонка:** `on_request=True`, если нормализованная подпись ценовой колонки содержит
`у.е.`, `у. е.`, `усл` или `y.e.`.

`detect_layout` вызывается **отдельно для каждого листа** — Optimus с тремя раскладками и
«выбрать все листы» работает корректно.

### 2. Парсер файла — `backend/app/catalog/parser.py`

Без структурных изменений: по-прежнему отдаёт `Rows` (список строк = списков ячеек) на лист.
`detect_layout` и `parse_rows` работают над `Rows`.

### 3. Парсинг строк — `backend/app/catalog/importer.py` (переписать `parse_rows`)

Новая сигнатура учитывает `data_start_row`, `manufacturer_col` и список ценовых колонок с флагом
`on_request`. Классификация каждой строки данных:

1. Собираем значения ценовых колонок. Ячейка-цена считается:
   - **числом** → реальная цена;
   - **«на запрос»** (`on_request`-колонка ИЛИ текст ∈ {`звоните`, `по запросу`, `уточняйте`, `прайс`, `запрос`}) → цена 0 + взвести `price_on_request` у позиции;
   - **пустой** → нет цены для этого уровня;
   - **нераспознанным числом** (похоже на число, но не парсится) → проблема «Цена не распознана».
2. Классификация строки:
   - **есть хоть одна реальная или on-request цена** → товар;
   - **цен нет, но есть Артикул/Код** → товар без цены → предупреждение «Нет цены»;
   - **цен нет и артикула нет, но есть текст** (в name_col, иначе manufacturer_col, иначе любая непустая описательная) → **строка-категория** → `current_category = текст`, товар не создаём;
   - **цен нет и текста нет** → разделитель → тихий пропуск (не в `problems`).

Дополнительно для товара:
- `manufacturer` = из `manufacturer_col`.
- `category` = `current_category` (если включён захват) → иначе значение `category_col` → иначе имя листа.
- `unit`: из `unit_col`; если значение содержит `/` (например `руб./шт`) — берём часть после последнего `/` и обрезаем валюту; по умолчанию `шт`.
- `article` = значение `article_col` (Артикул, иначе Код по приоритету определения).
- `price_on_request` (bool) — взводится, если хоть одна цена позиции «на запрос».

`ParsedRow` дополняется полями `manufacturer: str = ""`, `price_on_request: bool = False`.

### 4. Модель данных

- `CatalogItem` + `manufacturer: Mapped[str | None]` (миграция, nullable).
- `CatalogItem` + `price_on_request: Mapped[bool]` (миграция, `server_default=sa.text('false')`).
- `ColumnMapping` (Pydantic, `schemas.py`) + `manufacturer_col: int | None = None`,
  `data_start_row: int | None = None`. `header_row` уже есть.
- `import_parsed`: при создании/обновлении позиции пишет `manufacturer`, `price_on_request`;
  у.е./«звоните» цены пишутся как `ItemPrice.value = 0`. Ключ upsert — без изменений
  (`article`, иначе имя). При обновлении `price_on_request` переписывается значением из импорта.

### 5. API — `backend/app/catalog/router.py`

- **`POST /catalog/inspect`** (уже есть): для каждого листа дополнительно возвращает
  `detected: DetectedLayout | null` (Pydantic-схема `DetectedLayoutOut`), чтобы фронт предзаполнил маппинг.
- **`POST /catalog/import`**: принимает **по-листовый** подтверждённый маппинг
  (`sheets: list[{name, mapping: ColumnMapping}]`), каждый дефолтится из `detected`. Бэкенд по
  каждому листу применяет его маппинг: `parse_rows` → накопление в общий `import_parsed`
  (один прайс-лист/версия на импорт, как сейчас).

### 6. Фронтенд — мастер импорта (`frontend/src/pages/ImportPage.tsx`, `components/ColumnMapper.tsx`)

- После `inspect` мастер **предзаполняет** маппинг каждого листа из `detected` и показывает сводку:
  «лист X: Наименование=C, Код=B, Производитель=—, цены: РОЗН/ОПТ/ПАРТ».
- Пользователь подтверждает/правит роли и **привязывает найденные ценовые колонки к своим уровням
  цен** (`ColumnMapper`): автоподсказка по подписи (РОЗН→Розница, ОПТ→Опт, КР.ОПТ→…). Под у.е.-колонкой —
  пометка «цены по запросу».
- Многолистовые: листы **группируются по сигнатуре раскладки** (набор ролей+подписи цен);
  привязка уровней задаётся один раз на группу и применяется ко всем листам группы. Низкая
  `confidence` подсвечивается — просьба проверить вручную.
- Шаблон распознавания сохраняется у поставщика (`Supplier.column_mapping_template` уже есть) →
  следующий импорт того же поставщика подставляется сам.
- В таблице каталога (`CatalogPage.tsx`): показ `manufacturer` (если есть) и бейдж
  «уточнить стоимость» вместо «0», если `price_on_request`.

## Поток данных

1. Загрузка файла → `inspect` → по листам: колонки + `detected`.
2. Мастер предзаполняет маппинг, пользователь подтверждает + привязывает уровни цен.
3. `import` с по-листовыми маппингами → бэкенд: `detect`/`parse_rows` по листам → `import_parsed`
   (одна версия прайс-листа) → summary (создано/обновлено/цены/пропущено/проблемы).

## Обработка ошибок и краевые случаи

- `detect_layout` не нашёл заголовок → `detected=null`, мастер требует ручной маппинг (как сейчас).
- Низкая уверенность → подсветка, импорт не блокируется.
- «Нет цены» у позиции с артикулом → предупреждение в `problems` (как сейчас), позиция всё равно
  создаётся (без цены) — поведение сохраняем.
- Дубликат строки в файле — пропуск с предупреждением (как сейчас).
- у.е.-цены: значение 0 + `price_on_request` (конвертацию валют НЕ делаем).

## Вне рамок (YAGNI)

- Конвертация валют у.е.→рубли (импортируем 0 + отметка).
- Авто-определение типа (материал/работа) — `kind` по-прежнему выбирается в мастере.
- Захардкоженные профили поставщиков (подход C отклонён).
- Числовые/диапазонные фильтры по характеристикам (отдельная задача).
